# 基本設計

## 1. システム概要
本システムは複数店舗経営のアパレルショップ向けの在庫・発注管理を行うREST APIである。
店舗スタッフは自店舗の在庫を確認し、入荷登録および必要商品の発注を行うことができる。
※在庫の破棄・棚卸調整については、在庫変動履歴として将来的な拡張対象とする。
管理者は商品、店舗、在庫、発注状況を管理できる。

認証、認可にはKeycloakを利用し、OAuth2.0 / OpenID Connectにより発行されたJWTをSpring Boot API側で検証する。
利用者のロールに応じて、ADMINとSTORE_STAFFの権限制御を行う。
また、日次バッチにより在庫集計、在庫不足商品の抽出、発注候補商品の抽出を行う。

## 2. 利用者一覧
| 利用者 | 説明 |
|---|---|
| 管理者 | 商品・店舗・在庫・発注を管理する |
| 店舗スタッフ | 自店舗の在庫を確認し、入荷登録および発注を行う |

## 3. ロール一覧
| ロール | 説明 |
|---|---|
| ADMIN | 商品、ブランド、カテゴリ、店舗、全店舗在庫、全店舗発注、バッチ実行履歴を管理できる | 
| STORE_STAFF | 自店舗の在庫参照、入荷登録、自店舗発注登録、自店舗発注参照を行える |

管理者はシステム上では ADMIN ロールとして扱う。
店舗スタッフはシステム上では STORE_STAFF ロールとして扱う。

## 5. 業務フロー概要
### 5.1 管理者による商品・SKU登録フロー
1. 管理者はKeycloakでログインし、JWTを取得する。
2. 管理者は商品登録機能を使用し、商品名、ブランド、カテゴリ、シーズン、説明、販売状態などの商品情報を登録する。
3. 管理者は登録した商品に対して、SKU登録機能を使用し、SKUコード、カラー、サイズ、JANコード、価格などのSKU情報を登録する。
4. 同じ商品であっても、サイズやカラーが異なる場合は別SKUとして登録する。
5. 登録された商品・SKU情報は、在庫管理および発注管理で利用される。
6. 商品またはSKUの販売を停止する場合、物理削除は行わず、販売停止状態に変更する。

### 5.2 店舗スタッフによる在庫確認フロー
1. 店舗スタッフはKeycloakでログインし、JWTを取得する。
2. Spring Boot APIはJWTを検証し、利用者のロールと所属店舗情報を確認する。
3. 店舗スタッフは店舗別在庫一覧取得機能を使用し、自店舗の在庫一覧を確認する。
4. 在庫は店舗 × SKU単位で管理される。
5. 店舗スタッフは必要に応じて、商品名、SKUコード、カラー、サイズなどの条件で在庫を絞り込む。
6. 店舗スタッフが他店舗の在庫を参照しようとした場合、API側でアクセスを拒否する。
7. 管理者は全店舗の在庫一覧および在庫詳細を参照できる。

### 5.3 店舗スタッフによる入荷登録フロー
1. 店舗スタッフは自店舗に入荷した商品のSKUと入荷数を確認する。
2. 店舗スタッフは入荷登録機能を使用し、対象SKU、入荷数、入荷理由などを登録する。
3. Spring Boot APIは、ログインユーザーの所属店舗と入荷対象店舗が一致していることを確認する。
4. APIは対象店舗・SKUの在庫情報を取得する。
5. APIは在庫数を加算し、在庫変動履歴を登録する。
6. 在庫更新と在庫変動履歴登録は、同一トランザクションで実行する。
7. 在庫更新時は楽観ロックにより、同時更新による不整合を防止する。

### 5.4 店舗スタッフによる発注登録フロー
1. 店舗スタッフは自店舗の在庫一覧を確認する。
2. 店舗スタッフは在庫が不足している商品・SKUを確認する。
3. 店舗スタッフは発注登録機能を使用し、発注情報と発注明細を登録する。
4. 発注情報には、発注店舗、発注日、発注ステータスなどを含める。
5. 発注明細には、SKU、発注数量などを含める。
6. Spring Boot APIは、ログインユーザーの所属店舗と発注対象店舗が一致していることを確認する。
7. APIは発注情報と発注明細を同一トランザクションで登録する。
8. 登録直後の発注ステータスは、申請中とする。
9. 店舗スタッフは自店舗の発注一覧および発注詳細を確認できる。
10. 店舗スタッフが他店舗の発注情報を参照・登録しようとした場合、API側でアクセスを拒否する。

### 5.5 店舗スタッフによる発注キャンセルフロー
1. 店舗スタッフは自店舗の発注一覧からキャンセル対象の発注を選択する。
2. Spring Boot APIは、対象発注がログインユーザーの所属店舗の発注であることを確認する。
3. APIは、対象発注のステータスが申請中であることを確認する。
4. キャンセル可能な状態であれば、発注ステータスをキャンセルに更新する。
5. 承認済み、却下済み、完了済みの発注はキャンセルできない。
6. 店舗スタッフが他店舗の発注をキャンセルしようとした場合、API側でアクセスを拒否する。

### 5.6 管理者による発注ステータス更新フロー
1. 管理者はKeycloakでログインし、JWTを取得する。
2. 管理者は発注一覧取得機能を使用し、全店舗の発注状況を確認する。
3. 管理者は発注詳細取得機能を使用し、発注内容と発注明細を確認する。
4. 管理者は発注内容を確認したうえで、発注ステータス更新機能を使用する。
5. 管理者は発注ステータスを承認、却下、完了などに更新する。
6. STORE_STAFFは発注ステータス更新機能を利用できない。
7. ステータス更新時には、不正なステータス遷移を防止する。

ステータス遷移の例は以下とする。

申請中 → 承認済み
申請中 → 却下済み
承認済み → 完了
申請中 → キャンセル

### 5.7 日次バッチ処理フロー
1. 日次バッチは指定時刻に起動する。
2. 店舗別、SKU別の現在在庫を取得する。
3. 取得した在庫情報をもとに、日次在庫サマリを作成する。
4. 日次在庫サマリには、集計日、店舗、SKU、在庫数、安全在庫数、不足数などを保存する。
5. 在庫数が安全在庫数を下回る店舗・SKUを在庫不足商品として抽出する。
6. 抽出した在庫不足商品をもとに、発注候補データを作成する。
7. 発注候補データをCSV形式で出力する。
8. ローカル環境ではCSVをファイルとして出力し、AWS環境ではS3への出力を想定する。
9. バッチ名、開始時刻、終了時刻、実行結果、処理件数、エラー内容をバッチ実行履歴として保存する。
10. バッチ処理中にエラーが発生した場合、エラー内容をバッチ実行履歴に保存し、ログにも出力する。

### 5.8 認証・認可フロー
1. 利用者はKeycloakでログインする。
2. Keycloakは認証に成功した利用者に対してJWTアクセストークンを発行する。
3. 利用者はJWTを付与してSpring Boot APIを呼び出す。
4. Spring Boot APIはOAuth2 Resource ServerとしてJWTを検証する。
5. APIはJWT内のロール情報をもとに、ADMINまたはSTORE_STAFFの権限を判定する。
6. ADMINは管理者向け機能を利用できる。
7. STORE_STAFFは店舗スタッフ向け機能のみ利用できる。
8. STORE_STAFFが店舗IDを指定するAPIを利用する場合、APIはログインユーザーの所属店舗IDとリクエスト対象店舗IDを照合する。
9. 所属店舗と異なる店舗データへの参照・更新は拒否する。
10. アプリケーション側では利用者のパスワードを保持せず、KeycloakユーザーID、所属店舗ID、業務ロールなど業務上必要な情報のみを管理する。

## 6. 外部インターフェース概要
### 6.1 REST API
本システムは、店舗スタッフおよび管理者が利用するクライアントアプリケーションに対してREST APIを提供する。
APIはJSON形式のリクエストとレスポンスを使用する。
APIの詳細なエンドポイント、HTTPメソッド、リクエスト/レスポンスフォーマットは、API設計書に記載する。

### 6.2 認証基盤 Keycloak
本システムは認証・認可基盤としてKeycloakを利用する。
利用者はKeycloakでログインし、KeycloakがOAuth2.0 / OpenID Connectに基づいてJWTアクセストークンを発行する。

Spring Boot APIはOAuth2 Resource ServerとしてJWTを検証し、JWT内のロール情報をもとにアクセス制御を行う。
アプリケーション側では利用者のパスワードは保持しない。

### 6.3 データベース PostgreSQL
本システムは、業務データの永続化のためにPostgreSQLを使用する。
商品、SKU、店舗、在庫、在庫変動履歴、発注、発注明細、バッチ実行履歴などを管理する。

テーブル定義、主キー、外部キー、インデックス、制約については、DB設計書に記載する。

### 6.4 CSV出力
日次バッチ処理により、在庫不足商品をもとに発注候補CSVを出力する。
ローカル環境ではファイルシステムに出力し、AWS環境ではS3への出力を想定する。

CSVの出力項目、ファイル名、文字コード、出力タイミングについては、バッチ設計書に記載する。

### 6.5 ログ出力
本システムは、API実行時およびバッチ実行時にログを出力する。
ローカル環境ではコンソールログとして確認し、AWS環境ではCloudWatch Logsで確認することを想定する。

ログ出力方針、出力レベル、出力内容については、詳細設計書またはインフラ設計書に記載する。

### 6.6 外部公開方式
ローカル環境ではDocker Composeを使用し、Spring Boot API、PostgreSQL、Keycloakを起動する。
AWS環境では、Internet → ALB → ECS Fargate → RDS PostgreSQL の構成でAPIを公開することを想定する。

AWS環境では、DockerイメージをECRにpushし、ECS Fargate上でSpring Boot APIを実行する。


## 7. バッチ概要

### 7.1 売上取込バッチ

#### 目的
販売管理システムの売上データを取り込み、DBに保存すること。

#### 実行タイミング
販売管理システムから売上データが確定したタイミングでスケジューラにより起動する。  
ローカル環境では手動実行も可能とする。

#### 入力データ
- 売上CSV

売上CSVには、売上日、店舗コード、SKUコード、売上数量を含む。

#### 処理概要
1. 売上CSVファイルを取得する。
2. CSVの必須項目、日付、店舗コード、SKUコード、売上数量を検証する。
3. 検証済みの売上データを未反映状態でDBに保存する。
4. 同一CSVの二重取込を防止する。

#### 出力データ
- 売上テーブルへの保存
- バッチ実行履歴への保存

#### 異常時の扱い
- 売上CSVファイルの取得に失敗した場合、エラー内容をバッチ実行履歴に保存し、ログにも出力する。
- CSVの形式不正、必須項目不足、日付不正、店舗コード不正、SKUコード不正、売上数量不正がある場合、エラー内容をバッチ実行履歴に保存し、ログにも出力する。
- 売上テーブルへの保存に失敗した場合、エラー内容をバッチ実行履歴に保存し、ログにも出力する。
- 同一CSVの二重取込を検知した場合、取込済みとして扱い、在庫反映対象にはしない。

---

### 7.2 売上集計・在庫反映バッチ

#### 目的
DBに保存された未反映の売上データを店舗 × SKU単位で集計し、対象店舗・SKUの在庫数を減算する。  
また、売上反映後の在庫状態を日次在庫サマリとして保存する。

#### 実行タイミング
毎日深夜にスケジューラにより起動する。  
ローカル環境では手動実行も可能とする。

#### 入力データ
- 売上データ
- 店舗情報
- SKU情報
- 店舗別SKU在庫情報

売上取込バッチでDBに保存された未反映の売上データを取得することを想定する。

#### 処理概要
1. 売上取込バッチで保存された未反映の売上データを取得する。
2. 販売中のSKU情報を取得する。
3. 店舗 × SKU単位で現在在庫数を取得する。
4. 売上データをもとに、店舗 × SKU単位で売上数量を集計する。
5. 集計した売上数量を在庫数から減算し、店舗 × SKU単位で在庫数を更新する。更新時は楽観ロックにより同時更新を検知する。
6. 売上による在庫減算を在庫変動履歴として保存する。
7. 反映済みの売上データを処理済みに更新する。
8. 反映後の現在在庫数、安全在庫数、不足数を算出する。
9. 集計結果を日次在庫サマリとして保存する。

#### 出力データ
- 在庫テーブルへの更新
- 在庫変動履歴テーブルへの保存
- 売上データの処理済み更新
- 日次在庫サマリテーブルへの保存
- バッチ実行履歴への保存

#### 異常時の扱い
- 売上データの取得に失敗した場合、エラー内容をバッチ実行履歴に保存し、ログにも出力する。
- 対象店舗またはSKUが存在しない場合、エラー内容をバッチ実行履歴に保存し、ログにも出力する。
- 在庫情報の取得に失敗した場合、エラー内容をバッチ実行履歴に保存し、ログにも出力する。
- 在庫数の更新に失敗した場合、エラー内容をバッチ実行履歴に保存し、ログにも出力する。
- 在庫更新時に楽観ロックエラーが発生した場合、対象データをロールバックし、エラー内容をバッチ実行履歴に保存し、ログにも出力する。
- 日次在庫サマリの保存に失敗した場合、エラー内容をバッチ実行履歴に保存し、ログにも出力する。

---

### 7.3 発注候補CSV出力バッチ

#### 目的
売上反映後の現在在庫をもとに、安全在庫数を下回る店舗 × SKUを抽出し、発注候補CSVとして出力すること。

#### 実行タイミング
売上集計・在庫反映バッチ終了後にスケジューラにより起動する。  
ローカル環境では手動実行も可能とする。

#### 入力データ
- 店舗別SKU在庫情報
- 店舗情報
- 商品情報
- SKU情報

売上集計・在庫反映バッチ実行後の在庫テーブルから取得することを想定する。

#### 処理概要
1. 在庫テーブルから店舗 × SKU単位で現在在庫数、安全在庫数を取得する。
2. 在庫数が安全在庫数を下回る店舗 × SKUを抽出する。
3. 不足数をもとに発注候補数量を算出する。
4. 店舗、商品、SKU、カラー、サイズ、現在在庫数、安全在庫数、不足数、発注候補数量をCSVに出力する。

#### 出力データ
- 発注候補CSVファイル
- バッチ実行履歴への保存

#### 異常時の扱い
- 在庫情報の取得に失敗した場合、エラー内容をバッチ実行履歴に保存し、ログにも出力する。
- 商品情報またはSKU情報の取得に失敗した場合、エラー内容をバッチ実行履歴に保存し、ログにも出力する。
- 発注候補CSVの出力に失敗した場合、エラー内容をバッチ実行履歴に保存し、ログにも出力する。

---

### 7.4 バッチ共通仕様

#### 実行履歴
各バッチの実行開始時、正常終了時、異常終了時にバッチ実行履歴を保存する。  
保存内容は、バッチ名、開始時刻、終了時刻、実行結果、処理件数、エラー内容とする。

#### ログ出力
各バッチの開始、終了、処理件数、異常発生時のエラー内容をログに出力する。  
ローカル環境ではコンソールログとして確認し、AWS環境ではCloudWatch Logsで確認することを想定する。

#### 再実行方針
異常終了したバッチは、原因を解消したうえで手動再実行できるものとする。  
売上取込バッチでは、同一CSVの二重取込を防止する。  
売上集計・在庫反映バッチでは、未反映状態の売上データのみを処理対象とし、二重反映を防止する。

#### 日次在庫サマリの保存方針
日次在庫サマリは、売上反映後の在庫状態を日次履歴として保存するために作成する。  
発注候補CSV出力バッチでは、最新の在庫状態を参照するため、在庫テーブルを入力データとする。  
日次在庫サマリは、過去時点の在庫状態確認や在庫推移の分析に利用する。

日次在庫サマリは、在庫テーブルに存在する店舗 × SKU の組み合わせを対象に作成する。  
全店舗 × 全SKUを機械的に作成するのではなく、在庫管理対象として登録済みの在庫情報のみを保存する。

日次在庫サマリは履歴データとして保存するため、日々レコードが追加される。  
検索性能を考慮し、集計日、店舗ID、SKU IDにインデックスを設定する。  
保存期間は初期構築時点では制限しないが、将来的には一定期間経過後のアーカイブまたは削除を検討する。

### 8.2 主要テーブルの関係概要
商品、SKU、店舗、在庫、発注、売上取込、バッチ履歴がどのように関連するかを書く。

### 8.3 テーブル設計方針
- 商品本体とSKUを分離する
- 在庫は店舗 × SKU単位で管理する
- 入荷・売上による在庫増減は在庫変動履歴に保存する
- 売上CSV取込結果は売上取込テーブルに保存する
- 日次在庫サマリは履歴データとして保存する
- 物理削除ではなく、販売停止・無効化を基本とする
