## 1. セキュリティ設計方針

本システムでは、店舗スタッフ・店長・本部管理者が利用する業務APIと、来店客が利用するQR注文APIを分けてセキュリティを設計する。

店舗スタッフ、店長、本部管理者向けAPIでは、Amazon Cognito による OAuth2.0 / OpenID Connect 認証を利用する。
Cognito User Pool が JWT アクセストークンを発行し、Spring Boot API は OAuth2 Resource Server として JWT を検証する。

来店客向けAPIは Cognito 認証の対象外とする。
来店客はテーブル上の固定QRコードから注文画面へアクセスするが、QRコード自体を注文権限としては扱わない。
注文可否は、QRトークン、来店セッショントークン、テーブルステータス、来店セッション状態、有効期限によりサーバー側で判定する。

本システムのセキュリティ設計では、以下を基本方針とする。

| No  | 方針             | 内容                                                                  |
| --- | ---------------- | --------------------------------------------------------------------- |
| 1   | 認証方式の分離   | スタッフ系APIはJWT認証、来店客向けAPIはQR注文制御で扱う               |
| 2   | パスワード非保持 | アプリケーションDBの users テーブルにはパスワードを保持しない         |
| 3   | ロール認可       | ADMIN、STORE_MANAGER、STORE_STAFF の業務ロールで利用可能APIを制御する |
| 4   | 店舗単位認可     | 店舗スタッフ・店長は自店舗以外のデータを参照・更新できない            |
| 5   | サーバー側判定   | 画面表示やURLだけに依存せず、API側で認証・認可・注文可否を判定する    |
| 6   | 監査性           | スタッフ操作は、対象テーブルに監査項目が存在する場合、users.id を基に created_by / updated_by へ記録する |

## 2. 認証方式

### 2.1 店舗スタッフ・店長・本部管理者向け認証

店舗スタッフ、店長、本部管理者向けAPIでは、Amazon Cognito による OAuth2.0 / OpenID Connect 認証を利用する。
Cognito User Pool が JWT アクセストークンを発行し、Spring Boot API は OAuth2 Resource Server として JWT を検証する。

Spring Boot API ではフォームログイン、Basic認証、アプリケーション独自のパスワード認証は行わない。
ログイン画面、パスワード入力、パスワード検証、パスワードポリシーは Cognito 側で管理する。

業務APIへアクセスするクライアントは、Cognito から取得したアクセストークンを HTTP リクエストの Authorization ヘッダーに設定する。

```http
Authorization: Bearer {access_token}
```

認証が必要なAPIで Authorization ヘッダーが存在しない、Bearer形式ではない、JWTが無効、期限切れ、または検証に失敗した場合は `401 Unauthorized` を返す。

対象となるAPI区分は以下とする。

| API区分       | 認証方式        | 説明                                       |
| ------------- | --------------- | ------------------------------------------ |
| 店舗業務API   | Cognito JWT認証 | 店舗スタッフ・店長が利用する               |
| 本部管理API   | Cognito JWT認証 | 本部管理者が利用する                       |
| バッチ管理API | Cognito JWT認証 | 本部管理者、または運用管理者が利用する想定 |

### 2.2 来店客向けAPIの扱い

来店客向けAPIは Cognito 認証の対象外とする。
来店客は users テーブルでは管理せず、Cognito ユーザーとしても登録しない。

来店客はテーブル上の固定QRコードを読み取り、QRトークンを含むURLから注文画面へアクセスする。
ただし、固定QRトークンはテーブルを識別するための情報であり、固定QRトークン単体では注文権限として扱わない。

注文可否は、サーバー側で以下を組み合わせて判定する。

| No  | 判定項目               | 内容                                                   |
| --- | ---------------------- | ------------------------------------------------------ |
| 1   | QRトークン             | QRトークンに紐づくテーブルが存在すること               |
| 2   | 来店セッショントークン | 対象テーブルの有効な visitToken であること             |
| 3   | テーブルステータス     | テーブルが注文可能な状態であること                     |
| 4   | 来店セッション状態     | visit_sessions.status が ACTIVE であること             |
| 5   | 有効期限               | visit_sessions.expires_at が現在時刻より後であること   |
| 6   | メニュー状態           | 注文対象メニューが対象店舗に所属し、販売可能であること |

visit_sessions.expires_at は通常の注文可否を制御する主条件ではなく、放置された来店セッションや不正利用を防ぐための最大有効期限として扱う。
QRコードを第三者が読み取った場合でも、来店セッションが開始されていない、期限切れ、会計済み、またはテーブルが注文可能状態でない場合は注文を許可しない。

## 3. 認可方式

本システムでは、ロールベースアクセス制御と店舗単位アクセス制御を組み合わせる。

ロールベースアクセス制御では、ADMIN、STORE_MANAGER、STORE_STAFF の業務ロールに応じて利用可能APIを制御する。
店舗単位アクセス制御では、STORE_MANAGER および STORE_STAFF が自店舗以外のデータを参照・更新できないようにする。

Spring Boot API は、JWT の subject から CognitoユーザーIDを取得し、users テーブルを参照して業務ロールと所属店舗IDを取得する。
API認可では、JWTの有効性、業務ロール、所属店舗IDを組み合わせて判定する。

### 3.1 ロールベースアクセス制御

ロールベースアクセス制御では、APIごとに必要な業務ロールを定義し、ログインユーザーが必要ロールを満たす場合のみアクセスを許可する。

業務ロールはアプリケーションDBの users.role を正とする。
JWT内のロール情報は補助的に参照できるが、最終的な業務認可は users テーブルから取得した role を基に判定する。
Cognito Group と users.role が一致しない場合でも、業務認可では users.role を正とする。
Cognito Group は認証基盤側の補助情報として扱い、アプリケーション上の最終的な権限制御は users.role により判定する。

| ロール        | 主な利用者   | 利用可能範囲                                                 |
| ------------- | ------------ | ------------------------------------------------------------ |
| ADMIN         | 本部管理者   | 全店舗情報、全店舗売上、スタッフ管理、バッチ管理             |
| STORE_MANAGER | 店長         | 自店舗売上、自店舗メニュー管理、自店舗注文参照               |
| STORE_STAFF   | 店舗スタッフ | 自店舗のテーブル管理、注文確認、調理ステータス更新、会計処理 |

必要ロールを満たさない場合は `403 Forbidden` を返す。

### 3.2 ロール階層

本システムでは、上位ロールが下位ロールの権限を内包する。

| 上位ロール    | 内包するロール             |
| ------------- | -------------------------- |
| ADMIN         | STORE_MANAGER、STORE_STAFF |
| STORE_MANAGER | STORE_STAFF                |
| STORE_STAFF   | なし                       |

必要ロール欄に `STORE_STAFF` と記載されているAPIは、ロール階層により `STORE_MANAGER` および `ADMIN` も利用可能とする。
必要ロール欄に `STORE_MANAGER` と記載されているAPIは、ロール階層により `ADMIN` も利用可能とする。

ただし、ロール階層によってAPI利用権限を満たす場合でも、店舗単位アクセス制御の対象となるAPIでは所属店舗IDの判定を行う。

### 3.3 店舗単位アクセス制御

店舗単位アクセス制御では、STORE_MANAGER および STORE_STAFF が自店舗以外のデータを参照・更新できないようにする。

店舗業務APIでは、URL上の `storeId` と、JWT の subject から特定した users.store_id を比較する。
不一致の場合は `403 Forbidden` を返す。

```text
URLの storeId == users.store_id の場合のみアクセス許可
```

ADMIN は全店舗データにアクセスできるため、原則として店舗単位アクセス制御の制限を受けない。
ただし、ADMIN が店舗業務APIを代行操作する場合は、監査ログ上で操作ユーザーを識別できるようにする。

店舗単位アクセス制御の基本方針は以下とする。

| 対象ロール    | storeId 判定 | 内容                                            |
| ------------- | ------------ | ----------------------------------------------- |
| ADMIN         | 不要         | 全店舗データにアクセス可能                      |
| STORE_MANAGER | 必須         | users.store_id と URL の storeId が一致すること |
| STORE_STAFF   | 必須         | users.store_id と URL の storeId が一致すること |

URLに `storeId` を含まないAPIでも、対象データから店舗IDを特定できる場合は、取得したデータの store_id と users.store_id を比較する。
例として、注文ID、注文明細ID、会計ID、来店セッションIDを指定するAPIでは、対象データを取得した後に店舗所属を確認する。

## 4. JWT設計

### 4.1 JWTの発行元

JWTアクセストークンは Cognito User Pool が発行する。
Spring Boot API は JWT を発行せず、Cognito が発行したJWTを検証する OAuth2 Resource Server として動作する。

JWTの発行元は対象の Cognito User Pool とする。
Spring Boot API では issuer を検証し、想定した Cognito User Pool から発行されたトークンのみを受け入れる。

### 4.2 Spring Boot APIで検証する内容

Spring Boot API では、認証が必要なAPIに対して受け取ったJWTを検証する。
IDトークンはAPI認証には利用せず、API認証には access token を利用する。
APIでは token_use が `access` であることを確認し、スタッフ系APIで誤ってIDトークンを送信する実装ミスを防ぐ。

主な検証内容は以下とする。

| No  | 検証項目     | 内容                                                           |
| --- | ------------ | -------------------------------------------------------------- |
| 1   | 署名         | Cognito User Pool の公開鍵によりJWT署名が正しいこと            |
| 2   | issuer       | 想定した Cognito User Pool から発行されたJWTであること         |
| 3   | 有効期限     | exp が現在時刻より後であること                                 |
| 4   | 利用開始時刻 | nbf が設定されている場合、現在時刻が利用開始時刻以降であること |
| 5   | token_use    | access token として発行されたJWTであること                     |
| 6   | client_id    | access token の場合は `client_id` を検証し、想定した Cognito App Client に対して発行されたトークンであること |
| 7   | subject      | sub が存在し、users.cognito_user_id と紐づけ可能であること     |
| 8   | groups       | 必要に応じて cognito:groups から業務ロールを確認できること     |
| 9   | ユーザー状態 | users に該当ユーザーが存在し、deleted_at が NULL であること    |

JWTの署名、issuer、有効期限、token_use、client_id の検証に失敗した場合は `401 Unauthorized` を返す。
JWTは有効だが、users テーブルに該当する業務ユーザーが存在しない、論理削除されている、業務ロールや所属店舗IDの条件を満たさない場合は `403 Forbidden` を返す。

### 4.3 JWTから取得するユーザー情報

Spring Boot API は、JWTから最低限以下の情報を取得する。

| 項目              | JWT claim例             | 用途                                          |
| ----------------- | ----------------------- | --------------------------------------------- |
| CognitoユーザーID | `sub`                   | users.cognito_user_id と紐づける              |
| username          | `cognito:username` など | ログ出力、画面表示、監査補助                  |
| roles             | `cognito:groups`        | Cognito側のグループ情報確認、補助的な認可情報 |

店舗IDと業務ロールは、JWTだけに依存せず、JWTの subject から users テーブルを参照して取得する。

| 取得元 | 項目     | 用途                                             |
| ------ | -------- | ------------------------------------------------ |
| users  | id       | created_by / updated_by などの監査項目に利用する |
| users  | role     | 業務ロールによるAPI認可に利用する                |
| users  | store_id | 店舗単位アクセス制御に利用する                   |
| users  | deleted_at | 業務ユーザーが論理削除されていないことの判定に利用する |

users テーブルにパスワードは保持しない。
users.cognito_user_id と JWT の subject を紐づけて、アプリケーション側の業務ユーザーを特定する。

JWTから取得した情報と users テーブルの情報を組み合わせ、APIごとに以下を判定する。

| 判定項目     | 内容                                                   |
| ------------ | ------------------------------------------------------ |
| JWTの有効性  | Cognito が発行した有効なトークンであること             |
| 業務ユーザー | users.cognito_user_id に該当するユーザーが存在すること |
| 業務ロール   | APIに必要なロールを満たすこと                          |
| 所属店舗ID   | 店舗業務APIでは対象店舗へのアクセス権を満たすこと      |

## 5. Cognito設計

### 5.1 User Pool

User Pool は、店舗スタッフ、店長、本部管理者の認証基盤として利用する。
来店客はCognitoユーザーとして登録せず、来店客向けAPIはQRトークンと来店セッショントークンにより注文可否を制御する。

User Pool では、業務ユーザーのログインID、パスワード、パスワードポリシー、MFAなどの認証基盤側の設定を管理する。
アプリケーションDBの users テーブルにはパスワードを保持しない。

Spring Boot API は、User Pool が発行した access token を検証し、JWTの subject と users.cognito_user_id を紐づけて業務ユーザーを特定する。

### 5.2 App Client

App Client は、管理画面・業務画面から Cognito 認証を利用するために設定する。
スタッフ系APIでは、App Client を通じて発行された access token を Authorization ヘッダーで受け取る。

API認証ではIDトークンを利用せず、access token を利用する。
Spring Boot API では token_use が `access` であることと、`client_id` が想定した App Client であることを検証する。

### 5.3 Group

Cognito Group は、認証基盤側でユーザー種別を管理し、JWTにロール相当の情報を含めるために利用する。
ただし、店舗単位の業務認可や最終的な権限制御は、アプリケーションDBの users テーブルを正として判定する。
本システムでは以下の Group を定義する。

| Group名       | 対応する業務ロール | 説明                 |
| ------------- | ------------------ | -------------------- |
| ADMIN         | ADMIN              | 本部管理者           |
| STORE_MANAGER | STORE_MANAGER      | 店長                 |
| STORE_STAFF   | STORE_STAFF        | 店舗スタッフ         |

Cognito Group はJWTの `cognito:groups` に含まれる補助情報として扱う。
最終的な業務認可はアプリケーションDBの users.role を正として判定する。
Cognito Group と users.role が一致しない場合でも、APIの認可判定では users.role を優先する。

### 5.4 User

Cognito User は、店舗スタッフ、店長、本部管理者を表す認証ユーザーとして登録する。
来店客はCognito Userとして登録しない。

アプリケーションDBの users テーブルでは、Cognito User の subject を users.cognito_user_id として保持する。
これにより、Cognito の認証ユーザーとアプリケーション上の業務ユーザーを1対1で紐づける。

users テーブルでは、業務ロール、所属店舗ID、論理削除状態など、アプリケーションの認可に必要な情報を管理する。
ユーザーが退職、異動、権限変更された場合は、Cognito User と users テーブルの両方を運用上整合させる。
ただし、API認可の最終判定は users.role、users.store_id、users.deleted_at を基に行う。

## 6. 来店客向けQR注文セキュリティ

### 6.1 固定QRトークンの扱い

固定QRトークンは、客席テーブルを識別するための推測困難なランダム値とする。
QRコードには店舗IDやテーブルIDなどの連番値を直接埋め込まず、restaurant_tables.qr_token を埋め込む。

固定QRトークンはシステム内で一意とし、QRコードから対象店舗と対象テーブルを特定するために利用する。
ただし、固定QRトークン単体では注文権限として扱わない。

来店客がQRコードを読み取った場合でも、対象テーブルが注文可能状態であり、有効な来店セッションと visitToken が存在しなければ注文を許可しない。

| 項目       | 方針                                                   |
| ---------- | ------------------------------------------------------ |
| 生成方法   | 推測困難なランダム値として生成する                     |
| 一意性     | restaurant_tables.qr_token をシステム内で一意にする    |
| 用途       | QRコードから対象店舗・対象テーブルを特定する           |
| 権限扱い   | QRトークン単体では注文権限として扱わない               |
| 露出範囲   | テーブル上の固定QRコードとして来店客に公開される       |

### 6.2 来店セッションの扱い

来店セッションは、同一テーブルにおける来店客ごとの注文・会計単位を管理する業務データである。
HTTPセッションやCognitoログインセッションではない。

visitToken は、店舗スタッフがテーブル利用開始操作を行い、対象テーブルを AVAILABLE から OCCUPIED に変更するタイミングでサーバー側が発行する。
来店客は固定QRトークンに加えて visitToken を利用し、現在の来店セッションに対する注文であることを示す。

会計完了時は visit_sessions.status を CLOSED に更新し、ended_at を設定する。
これにより、過去に発行された visitToken を利用した追加注文を防止する。

visit_sessions.expires_at は、放置された来店セッションや不正利用を防ぐための最大有効期限として扱う。
通常の注文可否は table.status と visit_sessions.status を主軸に判定し、最終チェックとして expires_at が現在時刻より後であることも確認する。

| 項目       | 方針                                                               |
| ---------- | ------------------------------------------------------------------ |
| 発行契機   | テーブル利用開始時にサーバー側で発行する                           |
| 紐づき     | 店舗、テーブル、来店セッションに紐づける                           |
| 用途       | 来店単位の注文・会計を識別する                                     |
| 無効化     | 会計完了、期限切れ、来店セッション終了時に注文へ利用できなくする   |
| 保持方針   | 来店履歴としてDBに保持し、注文・会計と紐づける                     |

### 6.3 注文可能条件

来店客向け注文APIでは、Cognito認証を行わない代わりに、QRトークン、visitToken、テーブルステータス、来店セッション状態を組み合わせて注文可否を判定する。

注文を許可する主条件は、restaurant_tables.status が OCCUPIED であり、visit_sessions.status が ACTIVE であることとする。
会計待ち、清掃中、空席のテーブルでは注文を許可しない。

注文可能条件は以下とする。

| No  | 条件                                                       | 満たさない場合 |
| --- | ---------------------------------------------------------- | -------------- |
| 1   | qrToken に紐づくテーブルが存在すること                     | 注文不可       |
| 2   | visitToken に紐づく来店セッションが存在すること            | 注文不可       |
| 3   | qrToken のテーブルと visitToken の来店セッションが一致すること | 注文不可       |
| 4   | restaurant_tables.status が OCCUPIED であること            | 注文不可       |
| 5   | visit_sessions.status が ACTIVE であること                 | 注文不可       |
| 6   | visit_sessions.expires_at が現在時刻より後であること       | 注文不可       |
| 7   | 注文対象メニューが対象店舗に所属し、販売可能であること     | 注文不可       |

テーブルステータスごとの注文可否は以下とする。

| テーブルステータス | 注文可否 | 理由                                     |
| ------------------ | -------- | ---------------------------------------- |
| AVAILABLE          | 不可     | 空席であり、来店セッションが開始されていない |
| OCCUPIED           | 可       | 有効な来店セッションがある場合のみ注文可能 |
| PAYMENT_WAITING    | 不可     | 会計待ちであり、追加注文を受け付けない   |
| CLEANING           | 不可     | 清掃中であり、次の来店前の状態である     |

### 6.4 不正アクセス対策

来店客向けAPIでは、固定QRトークンが第三者に読み取られる可能性を前提に、QRトークン単体では注文できない設計とする。
注文登録時は、qrToken と visitToken の組み合わせが正しく、かつ対象テーブル・来店セッション・店舗が一致することをサーバー側で検証する。

不正アクセス対策は以下とする。

| No  | 対策                         | 内容                                                                 |
| --- | ---------------------------- | -------------------------------------------------------------------- |
| 1   | QRトークンの推測困難化       | 連番ではなくランダム値を利用し、URL推測によるアクセスを防止する     |
| 2   | QRトークン単体での注文禁止   | QRトークンのみでは注文登録を許可しない                               |
| 3   | visitToken検証               | 対象テーブルの有効な来店セッションに紐づく visitToken のみ許可する   |
| 4   | 状態チェック                 | OCCUPIED かつ ACTIVE の場合のみ注文を許可する                         |
| 5   | 期限チェック                 | expires_at を超過した来店セッションでは注文を許可しない               |
| 6   | 店舗・テーブル整合性確認     | qrToken、visitToken、注文メニューが同一店舗・同一来店に属することを確認する |

会計完了後、会計待ち、清掃中、空席のテーブルから送信された注文リクエストは状態不整合として拒否する。
過去の visitToken を再利用した注文や、別テーブルの qrToken と visitToken を組み合わせた注文も許可しない。

### 6.5 異常注文対策

来店客向け注文APIでは、正常な来店セッションであっても、大量注文や高額注文が発生する可能性を考慮する。
注文登録時に、注文数量、注文点数、注文金額の上限をチェックし、通常利用から外れる注文を異常注文として検知する。

異常注文対策は以下とする。

| No  | 対策               | 内容                                                             |
| --- | ------------------ | ---------------------------------------------------------------- |
| 1   | 注文数量上限       | 1明細あたりの quantity に上限を設ける                            |
| 2   | 注文明細数上限     | 1回の注文で登録できる明細数に上限を設ける                        |
| 3   | 注文金額上限       | 1回の注文合計金額が一定額を超える場合は注文を拒否または確認対象とする |
| 4   | 連続注文検知       | 短時間に同一 visitToken から大量の注文が送信された場合に検知する |
| 5   | 販売状態確認       | 販売停止中、売り切れ、論理削除済みメニューへの注文を拒否する     |

大量注文・高額注文を検知した場合は、入力値不正または状態不整合として注文登録を拒否する。
実装時は、業務要件に応じて上限値を設定し、必要に応じてログ出力やスタッフ確認の対象とする。
