## 1. API設計方針

本システムでは、SpringBootによりRESTfulなWebAPIを提供する。
クライアントは来店客向け注文画面、スタッフ向け注文管理画面、店長向け売上管理画面、本部管理者向け店舗管理画面の4種類があり、それぞれのクライアントから利用されるAPIを提供する。

API は JSON 形式でリクエスト・レスポンスを行う。
スタッフ・店長・本部管理者向けのAPIに関しては、Keycloak による認証済みの JWT を必要とする。

来店客向けAPIは Keycloak認証の対象外として、QRコード、来店セッショントークン、テーブルステータスにより注文可否を制御する。

## 2. 共通仕様

### 2.1 ベースURL

本システムのAPIは、以下のベースURLの配下に定義する。

| 環境         | ベースURL                      |
| ------------ | ------------------------------ |
| ローカル環境 | `http://localhost:8080/api/v1` |
| AWS環境      | `http://{alb-dns-name}/api/v1` |

APIのバージョン管理を行いやすくするため、URLに `/v1` を含める。

本ポートフォリオでは Route53 による独自ドメイン取得は対象外とし、AWS環境では ALB のDNS名を利用してAPIにアクセスする。
HTTPS化、独自ドメイン利用、CloudFront による静的ファイル配信は将来的な拡張対象とする。

来店客向けAPI、店舗業務API、本部管理API、バッチ管理APIは以下のパスで分類する。

| 区分          | パス                | 説明                                                                          |
| ------------- | ------------------- | ----------------------------------------------------------------------------- |
| 来店客向けAPI | `/customer`         | QR注文画面、メニュー取得、注文登録など、来店客が利用するAPI                   |
| 店舗業務API   | `/stores/{storeId}` | テーブル管理、注文確認、会計、店舗別メニュー・売上管理など、店舗単位で扱うAPI |
| 本部管理API   | `/admin`            | 店舗管理、スタッフ管理、全店舗売上参照など、本部管理者向けAPI                 |
| バッチ管理API | `/batches`          | バッチ実行、バッチ実行履歴確認などのAPI                                       |

### 2.2 認証方式

認証方式は 店舗業務API、本部管理APIは Keycloak による JWT 認証を採用する。
来店客向けAPIは Keycloak 認証の対象外とし、QRコード、来店セッショントークン、テーブルステータスにより注文可否を制御する。

### 2.3 権限制御方針

権限制御は、ユーザーの役割に応じてアクセス可能なAPIを制御する。
店舗業務API、本部管理APIでは、JWTに含まれるユーザー情報を基に権限を判定する。
来店客向けAPIでは、QRコード、来店セッショントークン、テーブルステータスによりアクセス制御を行う。

店舗業務向けAPIでは、JWTに含まれるユーザー情報を基に、店舗スタッフと店長の権限を判定する。店舗スタッフは自店舗のデータのみ操作可能とし、店長は自店舗の売上やメニュー管理も追加で可能である。

また、本部管理者向けAPIでは、全店舗のデータにアクセス可能であり、店舗管理やスタッフ管理も行える。

そして、本部管理者は、店長、店舗スタッフの権限も内包しており、店長は店舗スタッフの権限を内包している。

それぞれのAPIの詳細においては後述の「3. API一覧」「4. API詳細」にて、各APIの権限制御について明記する。

### 2.4 リクエスト形式

リクエストは JSON 形式で送信する。
必要に応じて、HTTPヘッダーに認証情報やコンテンツタイプを指定する。

### 2.5 レスポンス形式

レスポンスは JSON 形式で返却する。
成功時はデータを含むオブジェクトを返し、エラー時はエラー情報を含むオブジェクトを返す。

### 2.6 エラーレスポンス形式

エラー発生時は、以下の形式でエラー情報を返す。

```json
{
  "error": "エラーコード",
  "message": "エラーメッセージ"
}
```

### 2.7 HTTPステータス方針

| HTTPステータス            | 用途                                 |
| ------------------------- | ------------------------------------ |
| 200 OK                    | 取得・更新成功                       |
| 201 Created               | 登録成功                             |
| 204 No Content            | レスポンスボディなしの更新・削除成功 |
| 400 Bad Request           | 入力値不正                           |
| 401 Unauthorized          | 未認証                               |
| 403 Forbidden             | 権限不足・自店舗外データへのアクセス |
| 404 Not Found             | 対象データなし                       |
| 409 Conflict              | 楽観ロックエラー、状態不整合         |
| 500 Internal Server Error | 想定外エラー                         |

#### API-C-001 QR注文画面表示

| 項目           | 内容                                                                                                                                                                       |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| API ID         | API-C-001                                                                                                                                                                  |
| 機能名         | QR注文画面表示                                                                                                                                                             |
| HTTPメソッド   | GET                                                                                                                                                                        |
| エンドポイント | `/customer/tables/{qrToken}`                                                                                                                                               |
| 認証           | 不要                                                                                                                                                                       |
| 必要ロール     | なし                                                                                                                                                                       |
| 店舗制御       | QRトークン・来店セッションで制御                                                                                                                                           |
| 概要           | QRトークンから対象店舗・テーブルを特定し、注文可能状態と初期表示に必要なカテゴリ情報を取得する。テーブルが利用中で、有効な来店セッションが存在する場合のみ注文可能とする。 |

##### パスパラメータ

| パラメータ名 | 型     | 必須 | 説明           |
| ------------ | ------ | ---- | -------------- |
| qrToken      | string | Y    | 固定QRトークン |

##### クエリパラメータ

なし

##### リクエストヘッダー

なし

##### リクエストボディ

なし

##### レスポンス

```json
{
  "storeId": 1,
  "storeName": "渋谷店",
  "tableId": 10,
  "tableNumber": "A-1",
  "orderAvailable": true,
  "visitToken": "visit-token-sample",
  "categories": [
    { "categoryId": 1, "categoryName": "おすすめ" },
    { "categoryId": 2, "categoryName": "フード" }
  ]
}
```

##### 正常時HTTPステータス

200 OK

##### 主なチェック

| No  | チェック内容                                              | エラー時HTTPステータス |
| --- | --------------------------------------------------------- | ---------------------- |
| 1   | QRトークンが指定されていること                            | 400                    |
| 2   | QRトークンに紐づくテーブルが存在すること                  | 404                    |
| 3   | テーブルステータスが OCCUPIED であること                  | 409                    |
| 4   | 有効な来店セッションが存在し、status が ACTIVE であること | 409                    |
| 5   | 来店セッションの expires_at が現在時刻より後であること    | 409                    |

##### 主なエラー

| HTTPステータス | エラーコード     | 内容                   |
| -------------- | ---------------- | ---------------------- |
| 400            | VALIDATION_ERROR | 入力値が不正           |
| 403            | FORBIDDEN        | 注文権限がない         |
| 404            | NOT_FOUND        | 対象データが存在しない |
| 409            | CONFLICT         | 注文可能な状態ではない |
